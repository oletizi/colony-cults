# Design: Archive-Direct PDF Rendering (`impl:feature/archive-direct-pdf`)

- Date: 2026-07-17
- Roadmap item: `impl:feature/archive-direct-pdf`
- Status: designing (awaiting operator approval marker)
- Backend: `superpowers:brainstorming` via `/stack-control:design`; handoff to
  `/stack-control:define`.
- Mandate (operator-scoped): **PDF rendering only** — not the browser site, not a
  unified browser+PDF loader.

## Problem domain

`pdf:build` (shipped `corpus-print-pdf`, spec 007) renders print-native
facsimile-edition PDFs — verso facsimile scan │ recto FR OCR + EN translation (or
an english-only reading recto). It reads the **committed corpus snapshot**
(`site/data/<sourceId>.json.gz`), which is generated by the browser corpus loader
(`src/browser/load/books.ts` via `site:snapshot`).

That loader is **hard-coupled to Gallica's conventions**:

- It parses a `gallica.bnf.fr` ark out of each folio sidecar's `catalog_url`
  (`parseArkFromCatalogUrl`) and **fails loud on any other archive**
  (`archive.org/details/<id>` → *"catalog_url … with no parseable ark"*).
- The parsed ark is used **only** to construct Gallica **IIIF** image URLs (and a
  unit-ark vs page-ark consistency check) — it is not structural to page
  enumeration, which comes from the folios themselves.
- It maps page-range **extract** folios by absolute number: a Gallica extract of
  pages 48–50 has folios `f048–f050` but extract-relative translations
  `p001–p003`, so the loader derives page `p048` and can't find its translation.

This betrays the canonical-source-metadata model (spec 004), which is explicitly
**multi-archive** (Gallica / SLQ / IA / HathiTrust). As soon as sources arrive from
any non-Gallica archive — which they now do, via the museum (spec 011), page-range
(012), and archive.org (013) acquisition adapters — the PDF build breaks:

- **PB-P055** (de Groote *Nouvelle-France*, `archive.org/details/nouvellefrancec00groogoog`)
  — fully translated, but fails snapshot generation ("no parseable ark"). archive.org.
- **PB-P054** (Cour de cassation) — a **Gallica** page-range extract (pages 48–50);
  fully translated, but blocked by the folio(`f048`)↔translation(`p001`) numbering
  mismatch. Proves the defect is NOT archive-specific.
- **PB-P002** — was Gallica, re-acquired from archive.org; even a pin bump now breaks
  its existing snapshot.

The root cause is **per-source-archive special-casing where our own archive is
already uniform.** Every acquisition adapter resolves its source into ONE
normalized archive shape — `object_store` master key + `sha256`, per-page OCR +
translation text, and folio provenance — independent of the origin archive. The
loader ignores that uniform representation and re-derives source-archive specifics
(the Gallica ark, Gallica IIIF) instead. Reading exclusively from our archive
dissolves the friction by construction.

## Solution space

### Chosen — a PDF-scoped archive-direct edition reader

`pdf:build` reads **exclusively from our own normalized archive**, never from the
committed snapshot and never from source-archive URLs/arks. A new reader in
`src/pdf/load/` consumes the uniform archive provenance and produces the *same*
`Edition` view-model the Typst renderer already takes — so nothing downstream of
the reader changes.

**Pipeline:**
1. Resolve a source's archive directory from the bibliography SSOT.
2. Enumerate its folios (`fNNN.yml`) in order; that IS the page sequence.
3. Per page: read FR OCR + EN translation from the archive's text files; read the
   master-image location from the folio's `object_store` block.
4. **Images from `object_store` (B2) only** — fetch by `object_store.key`, verify
   `sha256`. No ark, no IIIF. A page whose master is absent from `object_store`
   **fails loud** (Constitution V) — an archive-completeness gap to fix, never a
   silent IIIF fallback (which would reintroduce the per-source friction).
5. **Ordering + folio↔translation correspondence** come from the source's *own*
   folio sequence, dissolving the extract-numbering bug (mapped within the source,
   not by a global page number).
6. **Untranslatable pages** honor an explicit archive marker: a *marked* page
   renders facsimile + FR OCR with a blank/"not translated" EN column; an
   *unmarked* missing translation still **fails loud** (keeps the safety net for
   genuine translation gaps).
7. **Reproducibility**: read a **pinned archive commit** and record it as the
   colophon `archiveRef` (same provenance guarantee as today, sourced from the
   archive clone rather than the snapshot pin sidecar).

Consequence (in-scope, PDF only): `pdf:build` requires the archive clone
(`CORPUS_ARCHIVE_PATH`) and no longer reads `site/data/*.json.gz`. This also
dissolves the snapshot-staleness / pin-bump friction. The browser's snapshot path
is untouched and out of scope.

**Reuse:** the general `@/archive` provenance readers (`readProvenance`, checksum,
`object_store`), the existing `b2`/object-store image provider and `sha256`
verification, the `@/pdf` `Edition` view-model + Typst renderer (unchanged), and
`@/bibliography` for source→directory resolution. Fail-loud, `@/` imports, no
`any`/`as`, files ≤ 300–500 lines.

### Rejected — add IA (then per-archive) support to the Gallica-coupled loader

Extend `books.ts` to also parse `archive.org` URLs, then HathiTrust, then the next
archive. Smallest immediate diff, but it treats the symptom: the coupling stays,
and every new archive needs another special case. It also keeps the PDF build
dependent on the shared snapshot-generation path. Rejected — the operator's
principle is to *dissolve* per-source friction, not enumerate it.

### Rejected — a unified shared corpus-loading layer for browser + PDF

Make one archive-agnostic loader both the browser site and `pdf:build` consume, so
the Gallica coupling can't diverge between surfaces again. Architecturally
appealing, but it couples two surfaces' evolution and is broader than the mandate.
Rejected explicitly by the operator: **scope restricted to PDF rendering.** (The
browser site keeps its snapshot path; a future unification, if ever wanted, is a
separate item.)

### Rejected — bump the archive pin + fix only snapshot generation

Refresh the corpus pin so the new sources snapshot, and fix the generation loader.
But generation is still Gallica-coupled (the same fixes as option 1), it keeps the
snapshot-staleness friction that surfaced here (a stale committed snapshot that
must be re-pinned and regenerated), and it re-derives, per build, data the archive
already holds. Rejected for the archive-direct read.

## Decisions

1. **Archive-direct read.** `pdf:build` renders exclusively from our normalized
   archive; it drops the committed-snapshot dependency (PDF only) and requires the
   archive clone.
2. **Images from `object_store` (B2) only** — no ark, no source-archive IIIF; a
   missing master fails loud (archive-completeness gap, not a fallback).
3. **Ordering + folio↔translation from the source's own folio sequence** —
   dissolves the page-range-extract numbering mismatch (PB-P054).
4. **Untranslatable pages: explicit archive marker** — marked → blank-EN render;
   unmarked missing translation → fail loud.
5. **Reproducibility via a pinned archive commit**, recorded in the colophon
   `archiveRef`.
6. **PDF-scoped** — the browser snapshot path is untouched and out of scope.

## Open questions

_Carry into `/stack-control:define`; none are blockers._

- **Untranslatable-marker representation.** What records "this page is deliberately
  untranslatable" in the archive — a sentinel translation file, a folio-sidecar
  flag, or a per-page provenance field? Whichever the translation pipeline can
  emit and the reader can read unambiguously.
- **Source → archive-directory resolution.** How the reader maps a `sourceId` to
  its archive directory (folio `local_path` prefix vs `object_store.key` prefix vs
  a bibliography field). Must be robust to the case/type/title-slug layout.
- **Reproducibility pinning mechanics.** Read `CORPUS_ARCHIVE_PATH` at HEAD and
  record its commit, vs a pinned worktree at a chosen ref (as `regen-snapshot.sh`
  does). Either way the colophon records exactly what was read.
- **Fate of the `iiif` image provider** in the PDF path. Retire it for
  archive-direct rendering (per "exclusively use the archive"), or keep it as an
  explicitly-invoked escape hatch for a not-yet-mirrored master. Leaning retire.
- **`english-only` variant + machine-assist labelling** must carry through the new
  reader unchanged (verify, don't redesign).
- **First end-to-end target.** PB-P055 (archive.org, fully translated) is the
  natural first proof; PB-P054 (Gallica extract) proves the numbering fix.

## Provenance

- Origin: interactive `/stack-control:design` (`superpowers:brainstorming`)
  session, 2026-07-17, on the long-lived PDF-generation branch, after
  `edition-publishing` shipped.
- Motivating context: attempting to build PDFs from newly-acquired sources
  (PB-P054/P055/P002) surfaced that a global snapshot pin-bump fails on
  `archive.org` `catalog_url` parsing and page-range-extract numbering — the PDF
  build's read path is Gallica-coupled.
- Operator steers that shaped the design: *"why are you obsessing over Internet
  Archive, when this is a general problem?"* (reframed from IA-support to
  Gallica-decoupling); *"restrict your mandate to PDF rendering"* (rejected the
  unified browser+PDF loader); *"since we have a well-understood, general archive
  of our own, we should exclusively use that — that should dissolve the per-source
  friction"* (the archive-direct principle); untranslatable-page handling = option
  (c), the explicit marker.
- Consumes the shipped acquisition adapters that normalize into the archive
  (`gallica-fetcher`, `source-group-acquisition`, `museum-acquisition-path`,
  `page-range-acquisition`, `archiveorg-acquisition-path`) and the shipped
  `corpus-print-pdf` build layer + `archive-object-store` (B2 masters).
